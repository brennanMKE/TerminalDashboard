## Description

`tuidash auto` (and bare `tuidash`) must start all three data sources, instantiate `AutoCoordinator`, render the tab bar shell with the active view's content, and handle all view-switching shortcuts.

## Expected behavior

- All three data sources start immediately at launch.
- `AutoCoordinator` subscribes to all three event streams.
- The root SwiftTUI view composes `TabBar` + the active view's content, switching the content area when the coordinator changes the active view.
- View-switching shortcuts (Cmd+Shift+G / C / L or their receivable equivalents) update the coordinator's active view and home view.
- Cmd+Shift+A toggles Auto Mode via the coordinator.
- Cmd+H opens `HelpOverlay` (auto mode variant — includes navigation section).
- q / Ctrl-C exits cleanly and terminates all three data sources and their subprocesses.
- Initial view: the first view for which config values are present, falling back to Git.

## Actual behavior

The `auto` subcommand stub prints a placeholder.

## Root cause

`AutoCommand.swift` contained only a stub that printed "auto: not yet implemented". No state objects, coordinator, or SwiftTUI view were wired up. `AutoDashboardView.swift` did not exist.

## Fix

**`AutoCommand.swift`** — fully implemented:
- Added `--config`, `--repo`, `--tool`, `--app`, `--crash-output`, `--subsystem`, `--category` (repeatable), `--process`, `--level` CLI flags matching the patterns from `GitCommand`, `CrashesCommand`, and `LogsCommand`.
- Config is loaded via `Config.load(override:)` then CLI flags are merged over config values for all three sections.
- `GitState`, `CrashesState`, `LogsState`, and `AutoCoordinator` are created on `MainActor.assumeIsolated`.
- Initial view is determined by which config section is present first (git → crashes → logs → git fallback).
- `AutoCoordinator.start(gitState:crashesState:logsState:)` wires up `onEvent` callbacks and starts all three data sources.
- `Application(rootView: AutoDashboardView(...)).start()` launches the TUI.

**`AutoDashboardView.swift`** — new file:
- `AutoDashboardView`: root view; `@ObservedObject` on coordinator, gitState, crashesState, logsState; conditional swap between `HelpOverlay` and `AutoDashboardContent` when `showHelp` is toggled.
- `AutoDashboardContent` / `AutoDashboardContentInner`: `GeometryReader` → `TabBar` (1 row) + content area (terminal height − 1). Content area switches on `coordinator.activeView`.
- `AutoGitContentInner`, `AutoCrashesContentInner`, `AutoLogsContentInner`: inline content views that replicate the dedicated-mode layouts and add `g/c/l/a/h` navigation commands to each view's command dispatcher. The navigation commands call `onSwitchView(_:)` and `onToggleAuto()` closures passed from the coordinator.
- Help overlay is the existing `HelpOverlay(isDedicatedMode: false, ...)` which shows the navigation section.

## Verification

`swift build` succeeds with zero errors under Swift 6 strict concurrency.

Running `swift run tuidash` (default subcommand = auto) and `swift run tuidash auto` both launch the full multi-view dashboard with TabBar, active-view content, and working `g/c/l/a/h` shortcuts.

## Files changed

- `Sources/TerminalDashboard/Commands/AutoCommand.swift` — replaced stub with full implementation
- `Sources/TerminalDashboard/Views/AutoDashboardView.swift` — new file; root view for auto mode

## Notes

Cmd+Shift key combinations are not receivable by SwiftTUI's single-character terminal input model. Navigation uses single lowercase letters (`g`, `c`, `l`, `a`, `h`) entered into the active view's command field instead. This is documented in `HelpOverlay`'s navigation section as "terminal permitting" and matches the pragmatic approach used for all other shortcuts.
