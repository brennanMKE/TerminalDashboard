## Description

`CrashesDataSource` must watch the two macOS crash log directories for new `.ips` and `.crash` files, match them against a configured app name filter, and emit `DashboardEvent` values.

## Expected behavior

- Watches `~/Library/Logs/DiagnosticReports/` and `/Library/Logs/DiagnosticReports/` using `DispatchSource` filesystem events, falling back to ≤5s polling if unavailable.
- Scans for existing files at launch.
- New file matching app filter (case-insensitive prefix): emits `critical`.
- Any other new file: emits `info`.
- Exposes a sorted list of matched and unmatched reports consumed by `CrashesView`.
- Works when one or both directories are absent.

## Actual behavior

No data source exists.

## Notes

Use `DispatchSource.makeFileSystemObjectSource` for the watch. The app filter is a case-insensitive prefix match against the filename. Expose enough metadata per file (name, date, process, exception type) for the view to render without re-reading every file on each tick.

Since macOS 12 the primary format is `.ips` (two concatenated JSON objects: header + payload). Older `.crash` text files may still appear. The same directory also holds `.hang`, `.spin`, `.cpu_resource`, and other non-crash diagnostics — the data source should include these in the unmatched section rather than ignore them. See [docs/collecting-crash-logs-macos.md](../docs/collecting-crash-logs-macos.md) for the full directory layout and file format details.
