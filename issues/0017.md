## Description

When a MacBook display goes to sleep — either from inactivity, lid close, or manual sleep — the three data sources continue polling and streaming unnecessarily. The tool should detect display sleep and suspend all active data sources, then resume them when the display wakes.

## Expected behavior

- On display sleep: all running data sources are suspended.
  - `GitDataSource` stops its polling loop.
  - `CrashesDataSource` pauses its filesystem watch and polling.
  - `LogsDataSource` stops the `log stream` subprocess.
- On display wake: all suspended data sources resume automatically.
  - Each data source restarts from its last known state.
  - `LogsDataSource` relaunches the `log stream` subprocess.
  - The UI reflects the resumed state within the normal poll interval.
- In auto mode, the `AutoCoordinator` coordinates suspend/resume across all three sources.
- In dedicated mode, the single active data source is suspended and resumed directly.

## Steps to reproduce

1. Run `tuidash` or any subcommand.
2. Let the display sleep (inactivity timeout, close lid, or `pmset sleepnow`).
3. Observe: data sources continue running unnecessarily in the background.

## Expected behavior

All data sources idle while the display is asleep and resume cleanly on wake.

## Actual behavior

Data sources run continuously regardless of display state.

## Notes

Use `IOKit` display sleep/wake notifications via `IORegisterForSystemPower` or the simpler `NSWorkspace.shared.notificationCenter` with `NSWorkspace.screensDidSleepNotification` and `NSWorkspace.screensDidWakeNotification`. The `NSWorkspace` approach is simpler and does not require IOKit directly — prefer it unless finer-grained control is needed.

Each data source needs `suspend()` and `resume()` methods (or equivalent). `GitDataSource` can cancel its polling `Task` and restart it on resume. `CrashesDataSource` can cancel the `DispatchSource` watchers and recreate them on resume. `LogsDataSource` can terminate the `Process` on suspend and relaunch it on resume.

A `SleepWakeMonitor` helper (separate file) should own the notification subscriptions and call into the coordinator or directly into the state objects. It must be `@MainActor` since it drives UI-adjacent state.

## Root cause

The three data sources had no concept of suspension. `GitDataSource` ran a polling `Task` indefinitely; `CrashesDataSource` kept its `DispatchSource` filesystem watchers and 5-second polling loop active; `LogsDataSource` left its `log stream` subprocess running. None of the runs reacted to display sleep, so battery, CPU, and (for `log stream`) subprocess overhead accumulated while the screen was off.

## Fix

Added a new `SleepWakeMonitor` (`Sources/TerminalDashboard/Coordinator/SleepWakeMonitor.swift`) — a `@MainActor` helper that subscribes to `NSWorkspace.shared.notificationCenter`'s `screensDidSleepNotification` and `screensDidWakeNotification`. It exposes `onSleep` and `onWake` closures that the owner wires up.

Each data source actor and its `@MainActor` `*State` wrapper gained `suspend()` and `resume()` methods:

- `GitDataSource` now tracks its polling `Task` so `suspend()` can cancel it; `resume()` starts a fresh polling task. Branch/files/etc. state is preserved across the suspension.
- `CrashesDataSource` tracks its watcher `Task` too; `suspend()` cancels the polling loop and tears down all `DispatchSource` watchers; `resume()` restarts the watcher loop and re-attaches sources. The initial rescan after resume emits events (using the preserved `knownFiles` set) so crash reports that appeared during sleep surface as new events.
- `LogsDataSource`'s `suspend()` terminates the `log stream` subprocess and cancels the reader task; `resume()` relaunches a fresh `log stream` invocation. Log lines emitted while suspended are lost (which matches what `log stream` itself would do).

`LogsState.resume()` previously meant "resume from user-initiated pause"; that method was renamed to `resumeFromPause()` so that `suspend()`/`resume()` have a consistent meaning across all three states (display sleep/wake).

`AutoCoordinator` gained `suspendAll()`/`resumeAll()` plus an `attachSleepWakeMonitor()` helper that wires a monitor to drive both. Each `*State` got an `attachSleepWakeMonitor()` helper too, so dedicated-mode commands can attach a monitor that controls a single source.

Wiring:
- `AutoCommand` calls `coord.attachSleepWakeMonitor()` after `coord.start(...)`.
- `GitCommand`, `CrashesCommand`, and `LogsCommand` each call `state.attachSleepWakeMonitor()` after `state.start()`.

The monitor instance is retained by the coordinator (auto mode) or state (dedicated mode) and lives for the lifetime of the process, which is exactly what's required since `Application.start()` calls `dispatchMain()` and never returns.

## Gotchas

- `LogsState.resume()` previously meant "resume from user-initiated pause" and was called by `togglePause()`. It was renamed to `resumeFromPause()` so the new `resume()` can mean "resume from display sleep" consistently with the other two states. No other call sites referenced the old method, so the rename was a contained refactor.
- `SleepWakeMonitor`'s observer tokens are not `Sendable`, so a nonisolated `deinit` cannot remove them. Since the monitor is retained for the process lifetime (`Application.start()` never returns), explicit teardown via `stop()` is documented but not required in normal use.
- The `CrashesDataSource` initial rescan after `resume()` uses `emitEvents: true` (versus `false` on first `start()`), relying on the persisted `knownFiles` set to ensure only files that appeared while suspended emit events.
