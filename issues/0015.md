## Description

`tuidash git`, `tuidash crashes`, and `tuidash logs` must each launch a full-screen SwiftTUI application with only their respective view and data source active. No tab bar, no AutoCoordinator, no view switching.

## Expected behavior

- `tuidash git`: starts `GitDataSource`, renders `GitView` full-screen.
- `tuidash crashes`: starts `CrashesDataSource`, renders `CrashesView` full-screen.
- `tuidash logs`: starts `LogsDataSource`, renders `LogsView` full-screen.
- Config is loaded from `.tuidash.toml` (or CLI flags) before the view starts.
- Cmd+H opens `HelpOverlay` (dedicated mode variant — no navigation section).
- q / Ctrl-C exits cleanly and terminates any background subprocesses.

## Actual behavior

Subcommands are stubs that print placeholders.

## Notes

Each subcommand creates its data source, passes it to the view, and calls `Application(rootView:).start()`. This is also where keyboard shortcuts for view-specific actions (`p`, `u`, `r`, `o`, `↑`, `↓`, `e`, `E`, `Space`, `c`) must be confirmed receivable via SwiftTUI's key event API — document the actual bindings used.

## Root cause

The three subcommand `run()` stubs printed placeholders and never launched a SwiftTUI `Application`. The `Config.load` and related helpers were implicitly `@MainActor`-isolated (inferred from context), blocking access from the nonisolated command `run()` context.

## Fix

Three dedicated root views were added (`GitDashboardView`, `CrashesDashboardView`, `LogsDashboardView`) — each wraps the existing dashboard layout and intercepts the `h` command in the `TextField` to toggle `HelpOverlay(isDedicatedMode: true, ...)`. Each command:

1. Calls `Config.load(override:)` (now `nonisolated`) to load configuration.
2. Merges CLI flag overrides (`--repo`, `--tool`; `--app`, `--crash-output`; `--subsystem`, `--category`, `--process`, `--level`) over config values.
3. Creates the appropriate `*State` on `MainActor` via `MainActor.assumeIsolated` (safe because ArgumentParser calls `run()` on the main thread).
4. Passes the state and config to the root view and calls `Application(rootView:).start()` inside `MainActor.assumeIsolated`, which calls `dispatchMain()` and never returns.

`Config.load`, `Config.parse`, and all private TOML helpers were marked `nonisolated` so they can be called freely from any context. `TOMLValue.string` and `TOMLValue.array` were also marked `nonisolated`.

### Keyboard bindings (actual)

All commands are entered into the `TextField` at the bottom of the view (`>` prompt) followed by Return:

| View | Key | Action |
|---|---|---|
| Git | `p` | Push (with diverge confirm if behind) |
| Git | `u` | Pull --ff-only |
| Git | `r` | Pull --rebase |
| Git | `o` | Launch external tool |
| Git | `h` | Toggle HelpOverlay |
| Crashes | `j` / `k` | Navigate down / up |
| Crashes | `e` | Extract selected report |
| Crashes | `E` | Extract all matched reports |
| Crashes | `h` | Toggle HelpOverlay |
| Logs | `p` | Toggle pause/resume |
| Logs | `c` | Clear ring buffer |
| Logs | `h` | Toggle HelpOverlay |

Ctrl-C exits via SwiftTUI's built-in SIGINT handler (`Application.stop()` → `exit(0)`).

`./build.sh build` completes with no errors or warnings under Swift 6 strict concurrency.
