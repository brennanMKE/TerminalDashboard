## Description

`CrashesView` renders matched and unmatched crash reports from `CrashesDataSource`, with a detail pane for the selected report and key bindings for navigation and extraction.

## Expected behavior

- Matched reports section: filename, crash date, process, exception type; newest first; highlighted/bold.
- Unmatched reports section: same columns, dimmed, separated by a divider.
- Detail pane for selected report: signal, exception type, first lines of backtrace.
- Selection moves with `↑` / `↓`.
- `e` extracts the selected crash file to the configured output directory (creates directory if absent; confirms path in footer).
- `E` extracts all matched crashes.
- Footer shows extraction confirmation or error messages. No persistent key hints.
- "Not configured" prompt shown if no `app` filter is set.

## Actual behavior

No view exists.

## Notes

The detail pane reads the first N lines of the selected `.ips` or `.crash` file on demand — don't pre-read all files at load time. The layout needs to split available vertical space between the list and the detail pane; a fixed split (e.g. 60/40) is fine for v1.

`.ips` files are two concatenated JSON objects (header line + payload) — the detail pane should show the exception type, signal, and the first few frames of the faulting thread parsed from the payload. The unmatched section should surface `.hang`, `.spin`, and `.cpu_resource` files as well, not just `.crash`/`.ips`. See [docs/collecting-crash-logs-macos.md](../docs/collecting-crash-logs-macos.md) for format details and the full list of diagnostic extensions.
